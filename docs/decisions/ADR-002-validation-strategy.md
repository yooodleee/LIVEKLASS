# ADR-002 유효성 검증 전략

## 결정 배경

3단계 폼에서 언제, 어떤 범위로 유효성 검증을 실행할지 결정이 필요했다. 전체 검증을 제출 시 한 번만 실행하는 방식과 스텝별로 분리하는 방식 사이의 UX 차이가 크다.

## 선택한 방식

- **스텝 전환 시**: `trigger(스텝에 해당하는 필드 목록)`으로 현재 스텝 전체 부분 검증
- **필드 blur 시**: `trigger(fieldName)` 또는 `mode: 'onBlur'`로 개별 필드 검증
- **제출 시**: `stepThreeSchema`(전체 통합 스키마)로 최종 검증
- **스키마 분리**: `stepOneSchema` / `stepTwoSchema` / `stepThreeSchema`로 파일 분리

## 고려한 대안

- **제출 시 전체 검증만**: 구현은 단순하나 사용자가 마지막 스텝에서야 오류를 인지하는 UX 문제
- **onChange마다 검증**: 즉각적인 피드백이지만 입력 중 에러가 표시되어 불필요한 스트레스 유발

## 선택 이유

스텝 전환 시 부분 검증은 사용자가 각 스텝을 완료하는 시점에 즉시 피드백을 받아 오류를 빠르게 수정할 수 있다. blur 시 개별 검증은 입력 완료 후에만 에러를 표시하여 불필요한 에러 노출을 방지한다.

## 트레이드오프

- **장점**: 스텝 단위 빠른 피드백, blur 기반으로 UX 스트레스 최소화, 제출 시 최종 안전망 확보
- **단점**: 스키마를 스텝 단위로 분리 관리해야 하는 추가 유지보수 비용
